# Nuvio v1.0 Vision

## What nuvio is

nuvio is the **visual editing layer** for **React + Vite + Tailwind** apps. It runs in **development only** and writes changes back to real source files — not a hosted builder, not a production runtime.

## Who it is for

**Vibe coders and frontend developers** who:

- Build with React, Vite, and Tailwind
- Use AI for scaffolding but waste tokens on repetitive UI tweaks
- Want to click, edit, preview, and apply — without leaving localhost

## Problem it solves

AI coding tools are inefficient for low-intelligence visual work: padding, colors, spacing, text tweaks, card layout. Each small change costs tokens, time, and retries.

nuvio replaces that loop with direct visual editing on localhost, synchronized to JSX/Tailwind source.

**Positioning:** Use AI for architecture and coding. Use nuvio for visual refinement.

## Success at v1.0

A React + Vite + Tailwind developer can:

```bash
pnpm dlx @nuvio/cli init --yes
pnpm dev
```

Maximum **Vite + Tailwind** coverage: [COVERAGE.md](COVERAGE.md).

Then:

1. Turn **Edit on** in the nuvio chip
2. **Click** an element (tagged or via click-to-tag)
3. **Edit** text and Tailwind styles visually
4. **Preview** human-readable changes
5. **Apply** safely to source files
6. **Undo** the last apply in-session

…with **minimal setup**, clear errors when something is unsupported, and `nuvio doctor` / `nuvio scan` when setup is wrong.

# v0.6 — Click To Tag

## Goal

Remove the need for manual `data-nuvio-id` creation. Users click any visible element and nuvio tags it in source automatically.

## Problem today

After `nuvio init`, only the starter `page.title` (or manually tagged elements) are editable. New users hit a wall:

```jsx
<h1 data-nuvio-id="page.title">Welcome</h1>
```

Most AI-generated JSX has **no ids**. Users must read docs, pick id names, edit files, restart dev — high friction for vibe coders.

## Features

| Feature | Description |
| ------- | ----------- |
| Detect untagged elements | Overlay knows click target is not in source index |
| **Make Editable** action | User confirms tagging an untagged element |
| Auto-generate ids | Stable, human-readable id from element role + context (e.g. `hero.title`, `card.metrics.label`) |
| Source insertion | AST patch adds `data-nuvio-id` to the correct JSX opening element |
| Reindex | Vite plugin rescans; index updates |
| Refresh overlay | Selection becomes fully editable without full page reload |

## User flow

```text
Edit On
  → Click untagged element
  → Overlay: "This element isn't editable yet"
  → [Make Editable]
  → nuvio inserts data-nuvio-id in source
  → Reindex + refresh
  → Element appears in task menu / style controls
  → User edits → Preview → Apply
```

## Telemetry

| Event | When |
| ----- | ---- |
| `tag_element_started` | User confirms Make Editable |
| `tag_element_completed` | Id inserted + reindex ok |
| `tag_element_failed` | AST/index/write failure (safe `reason` enum) |

Privacy: same rules as existing telemetry — no paths, no element text in payload.

## Acceptance criteria

- [x] Untagged click shows clear CTA (not a dead click or cryptic error)
- [x] Generated id is unique; collision triggers rename suggestion before write
- [x] Id is inserted on the **correct** JSX node (not a parent/child mismatch)
- [x] Works for common elements: `h1`–`h6`, `p`, `span`, `button`, `div` with text, `td`/`th`
- [x] Fails closed for: fragments without host, map loops without stable key context, third-party components that don't forward attributes
- [x] Reindex picks up new id within one dev cycle (HMR or explicit refresh)
- [ ] Tag + edit + apply works on fresh `pnpm create vite` react-ts template without manual ids (except optional starter) — manual dogfood pending
- [x] Telemetry events fire; opt-out respected
- [x] Unit tests for id generation, AST insert, duplicate handling

## Technical design

### Overlay

- On select: if id not in index → show **Make Editable** panel state (not standard editor)
- RPC: `tagElement { file, line, column, suggestedId }` or `tagElement { domPath, suggestedId }`

### Vite plugin

- New RPC handler `tagElement`:
  1. Resolve JSX node from source location (existing index metadata + Babel)
  2. Validate node is taggable (opening element, not spread-only component)
  3. Call ast-engine `insertNuvioId(opening, id)`
  4. Write file, format if configured
  5. Trigger reindex
  6. Return `{ ok, id, file }` — file path only in dev response, never in telemetry

### AST engine

- New op: `insertDataNuvioId` on `JSXOpeningElement`
- Id validator: `^[a-z][a-z0-9]*(\.[a-z][a-z0-9]*)+$` (segmented names)
- Refuse if attribute already present

### Id generation heuristic

Priority:

1. Existing semantic hints (`aria-label`, `role`, heading level → `section.title`)
2. Parent id prefix + child role (`metric.card` → `metric.card.label`)
3. Component file name + ordinal (`dashboard.title`, `dashboard.title2`)

Never use: user text content, file paths, package names.

### Implementation tasks

| # | Task | Package |
| - | ---- | ------- |
| 1 | `insertDataNuvioId` AST helper + tests | `ast-engine` |
| 2 | `tagElement` RPC + file write + reindex | `vite-plugin` |
| 3 | Untagged selection UI + Make Editable | `overlay` |
| 4 | Id suggestion utility (pure function) | `shared` or `ast-engine` |
| 5 | Overlay telemetry events | `overlay` + docs |
| 6 | Dogfood: tag elements on demo-app + TailAdmin without pre-placed ids | `apps/*` |
| 7 | Update `nuvio/AGENT.md` template + user docs | `cli` templates, `docs/` |

## Risks

| Risk | Mitigation |
| ---- | ---------- |
| Wrong JSX node tagged (wrapper components) | Require resolved opening element; fail with "add forwardRef/data-nuvio-id to wrapper" |
| Duplicate ids in map loops | Detect `map(` parent; require per-item id strategy or block with explanation |
| shadcn Radix wrappers don't forward attributes | Document limitation; v0.8 addresses library-specific paths |
| Id generation ugly/unstable | Show suggested id editable before confirm |
| Breaks user formatting | Run formatter hook or preserve existing quote style |

## Estimated complexity

**Large (L)** — ~3–4 weeks focused work across overlay, vite-plugin, ast-engine, tests, and docs. Highest ROI milestone for v1.0.

# v0.7 — Better Tailwind Support

## Goal

Support modern Tailwind codebases — the way real projects write `className`, not only string literals.

## Current limitations

| Pattern | Today |
| ------- | ----- |
| `className="p-4 text-sm"` | Supported |
| `className={cn("p-4", "rounded")}` | Supported (string-literal args only) |
| `className={cn("p-4", isActive && "bg-blue-500")}` | **Rejected** |
| `className={clsx(...)}` with variables | **Rejected** |
| `className={cva(...)}` | **Rejected** |
| Template literals | **Rejected** |
| Arbitrary values `text-[#fff]` | **Rejected** (whitelist) |
| Tailwind v4 syntax edge cases | Partial |

## Support targets

### Phase A — Expression modes

| Mode | Patterns |
| ---- | -------- |
| `cn-conditional` | `cn("base", condition && "token")` — merge into base string or named slot |
| `classnames` | `classnames("a", { active: isActive })` — static key map only |
| Read-only preview | Show computed class intent even when patch targets one branch |

### Phase B — Investigate

| Library | Approach |
| ------- | -------- |
| `cva()` | Parse variant keys; patch default variant string or one variant branch |
| `tailwind-variants` | Same; map overlay presets → variant keys |
| `twMerge` in userland | Already used in engine; ensure v4 token parity |

### Visual editing categories (Simple Mode)

Expand Quick Style coverage:

| Category | Examples |
| -------- | -------- |
| Typography | size, weight, leading, align |
| Colors | text, bg, border (palette swatches) |
| Spacing | padding, margin, gap |
| Borders | width, color, radius |
| Shadows | sm, md, lg |
| Sizing | w, h, max-w, min-h |

## Acceptance criteria

- [x] `cn` with one conditional static token: preview + apply updates correct branch or flattened result
- [x] `clsx` / `classnames` static object map: merge/remove utilities without breaking JS logic
- [x] Reject unsupported patterns with plain-language reason
- [ ] Tailwind v4 common tokens in whitelist (align with `tailwind-v4-test` app)
- [x] No silent wrong-file or wrong-node writes
- [x] Regression suite: literal, cn-basic, cn-conditional fixtures
- [ ] Dogfood: edit TailAdmin card styles written with `cn()` in real components

## Technical design

### AST engine

- Extend `ClassNameMode` enum: `literal` | `cn-basic` | `cn-conditional` | `classnames-static`
- Per-mode **read binding** and **write binding** (already started in `getClassNameBinding`)
- Patch ops unchanged at overlay boundary (`mergeTailwindClassName`); engine resolves target expression

### Overlay

- Style controls emit same patch ops; engine picks mode from indexed metadata
- Index enriches entries: `classNameMode`, `editableBranches[]`

### Vite plugin / index

- At index time, classify `className` expression shape per host
- Store mode in source index JSON sent to overlay

## Risks

| Risk | Mitigation |
| ---- | ---------- |
| Conditional patches break runtime logic | Only patch branches with statically known string literals |
| Explosion of edge cases | Fail-closed; expand modes incrementally with fixtures |
| Tailwind v4 drift | Dedicated `apps/tailwind-v4-test` gate in CI |
| cva/tv too complex for v0.7 | Defer full cva to v0.8 if needed; document |

## Implementation tasks

| # | Task | Package |
| - | ---- | ------- |
| 1 | `cn-conditional` read/write in ast-engine | `ast-engine` |
| 2 | `classnames-static` mode | `ast-engine` |
| 3 | Index `classNameMode` metadata | `vite-plugin` |
| 4 | Expand tailwind whitelist + v4 tokens | `ast-engine` |
| 5 | Overlay: no UX change if patch contract stable | `overlay` |
| 6 | SIMPLE_MODE audit for new error messages | `overlay`, docs |

## Estimated complexity

**Large (L)** — ~3–5 weeks. Depends on how far cva/tv goes in v0.7 vs v0.8.

# v0.8 — Component Library Support

## Goal

Support real-world React projects beyond hand-instrumented demos — starting with the libraries vibe coders actually use.

## Priority order

| Priority | Library | Rationale |
| -------- | ------- | --------- |
| 1 | **shadcn/ui** | Dominant in AI-generated React apps; copy-paste components |
| 2 | **TailAdmin** | Already dogfooded (`apps/tailadmin-dogfood`) |
| 3 | **DaisyUI** | Common Tailwind component layer |

### Optional (post-v1.0 or stretch)

Flowbite, Mantine, Chakra, Material UI — different styling models; lower ROI for Tailwind-first positioning.

## Component detection

| Signal | Use |
| ------ | --- |
| Import path (`@/components/ui/button`) | shadcn detection |
| File path conventions (`components/ui/`) | Auto-suggest taggable hosts |
| `forwardRef` + `Slot` patterns | Radix primitives — patch outer host or `className` merge target |
| Known component registry | Map `Button`, `Card`, `Table` → editable regions |

## Editable regions strategy

| Pattern | Strategy |
| ------- | -------- |
| shadcn `Card` | Tag `Card`, `CardHeader`, `CardTitle`, `CardContent` separately |
| shadcn `Button` | Text child or `children` string; `className` on `Button` |
| shadcn `Table` | Reuse v0.5 table contract (`orders.row.{id}.nameText`) |
| TailAdmin wrappers | `data-nuvio-id` forward through `TableCell`, `TableRow` (done in dogfood) |
| DaisyUI | `className` on native element inside component |

## Patch strategy

1. **Prefer leaf hosts** with literal or cn-basic `className`
2. **Slot components:** patch `className` prop on the call site, not inside `node_modules`
3. **Compound components:** index each export as separate id host
4. **Fail closed** on opaque third-party roots

## Supported components (v1.0 target)

### shadcn/ui (minimum)

- Button, Card (+ Header/Title/Description/Content), Input, Label, Badge, Table, Dialog (title only)

### TailAdmin (minimum)

- Dashboard cards, metrics, charts titles, Recent Orders table, sidebar nav labels, form labels

### DaisyUI (minimum)

- `btn`, `card`, `table`, `navbar` — class-based hosts on native elements

## Limitations (document honestly)

- Radix portals (Dialog content in portal) — edit trigger/label only unless portal host tagged
- Data grids with virtualized rows — row ids must use stable keys
- MUI/Chakra — styled-system props not Tailwind patches

## Acceptance criteria

- [ ] Fresh Vite + shadcn template: click-to-tag + edit button label + card padding applies correctly
- [x] TailAdmin dogfood: library detected + instrumented dashboard edits (manual)
- [ ] DaisyUI demo: edit button + card title on official starter
- [x] Library detected in dev diagnostics (`detectedLibraries` — `nuvio scan` deferred to v0.9)
- [x] Unsupported component shows helpful message (shadcn wrapper guidance)
- [x] No patches inside `node_modules`

## Technical design

### Shared library registry

```ts
// packages/shared/src/library-registry.ts (conceptual)
type LibraryProfile = {
  id: "shadcn" | "tailadmin" | "daisyui";
  detect(project): boolean;
  hostPatterns: HostPattern[];
  forwardPropComponents: string[];
};
```

### Vite plugin

- Extend source index with `libraryHints[]`
- Optional import graph scan (lightweight — only `components/ui`)

### Overlay

- Task menu variants per library (Card vs btn vs table)
- "Make Editable" uses library-aware id suggestions (`card.stats.title`)

## Risks

| Risk | Mitigation |
| ---- | ---------- |
| shadcn copies differ per project | Detect by path convention, not package name |
| Radix Slot className merging | Patch call-site `className` on shadcn wrapper |
| Maintenance burden | Start with 8–10 components; expand post-v1.0 |
| Click-to-tag on wrong node in compound components | Library-specific disambiguation rules |

## Implementation tasks

| # | Task | Package |
| - | ---- | ------- |
| 1 | Library detection + registry | `shared`, `vite-plugin` |
| 2 | shadcn host patterns + fixtures | `ast-engine`, `apps/shadcn-example` |
| 3 | TailAdmin: generalize dogfood patches | `apps/tailadmin-dogfood` |
| 4 | DaisyUI starter fixture | `apps/` or `examples/` |
| 5 | Overlay task routing per library | `overlay` |
| 6 | Docs: per-library setup guides | `docs/` |

## Estimated complexity

**Extra large (XL)** — ~4–6 weeks. Parallelizable after v0.6/v0.7 foundations.

# v0.9 — Doctor, Scan, Stats

## Goal

Improve setup validation and project visibility — reduce "nuvio doesn't work" support burden.

## Commands

```bash
nuvio doctor    # Is this project wired correctly?
nuvio scan      # What's editable? What's missing?
nuvio stats     # Project-level summary for humans + telemetry
```

## `nuvio doctor`

**Purpose:** Pass/fail checklist after init or when debugging.

### Example output

```text
nuvio doctor — MyApp

  ✅ @nuvio/vite-plugin installed
  ✅ @nuvio/overlay installed
  ✅ vite.config.ts contains nuvio()
  ✅ optimizeDeps.exclude includes @nuvio/overlay
  ✅ main.tsx imports @nuvio/overlay/style.css
  ✅ NuvioDevShell mounted in app root
  ⚠ Tailwind not detected — style edits may not apply visually
  ❌ No data-nuvio-id found — run dev and use Make Editable, or add ids manually
  ✅ Dev server reachable (localhost:5173)

Result: 7/9 passed — nuvio partially ready
```

Checks: package.json deps, vite config AST sniff, main entry import, app shell grep, tailwind detect, index count (via plugin ping or offline scan), optional dev-server health.

## `nuvio scan`

**Purpose:** List indexed editable regions (offline or via dev session).

### Example output

```text
nuvio scan — 24 editable hosts

  page.title          src/App.tsx:12
  hero.subtitle       src/components/Hero.tsx:28
  metric.orders.label src/components/Metrics.tsx:41
  …

  3 untagged candidates near click targets (use Make Editable in browser)
  1 duplicate id: nav.home (src/Nav.tsx, src/Footer.tsx) — fix before apply
```

## `nuvio stats`

**Purpose:** Short summary for maintainer / optional anonymous telemetry batch.

### Example output

```text
nuvio stats

  Editable hosts:     24
  Tagged files:       8
  Libraries detected: shadcn, tailwind v4
  Table hosts:        2
  Duplicate ids:      0
  Class modes:        literal 18, cn-basic 6
```

### Telemetry (optional, privacy-safe)

| Event | Properties |
| ----- | ---------- |
| `doctor_run` | pass count, fail count, warning count (no paths) |
| `scan_run` | host count, duplicate count, library enum |
| `stats_run` | aggregate counts only |

Never send: file paths, project name, host id strings in bulk.

## Acceptance criteria

- [x] `nuvio doctor` exits 0 when fully wired, non-zero when critical checks fail
- [x] `nuvio scan` works without browser (offline AST scan) and matches live index
- [x] `nuvio stats` human-readable; suitable for `--json` flag (maintainers)
- [x] All commands respect `NUVIO_TELEMETRY=0`
- [x] Commands run in <5s on medium projects
- [x] Documented in `nuvioUser.md` and CLI `--help`

## Technical design

### CLI structure

```
packages/cli/src/

  scan.ts        # reuse ast-engine / vite-plugin scan logic (extract shared scanner)
  stats.ts       # aggregate scan results
```

Extract shared **project scanner** from `detect-project.ts` + lightweight id walk (reuse vite-plugin index builder in offline mode).

### Vite plugin

- Optional HTTP/WS `getIndexStats` for live mode when dev server running
- Offline: CLI imports `@nuvio/vite-plugin/scan` or shared scanner package

## Risks

| Risk | Mitigation |
| ---- | ---------- |
| Offline scan diverges from live index | Same parser code path; fixture tests |
| Doctor false positives on custom vite configs | Allow `--cwd`, document edge cases |
| Stats feels like spying | Opt-out, aggregates only, no path telemetry |

## Implementation tasks

| # | Task | Package |
| - | ---- | ------- |
| 1 | Extract shared source scanner | `shared` or `vite-plugin` |
| 2 | `doctor` command + checks | `cli` |
| 3 | `scan` command + table output | `cli` |
| 4 | `stats` command + `--json` | `cli` |

| 6 | Tests with fixture projects | `cli/test` |

## Estimated complexity

**Medium (M)** — ~2–3 weeks after scanner extraction.

# v1.0 — Stable Release

## Goal

Production-ready **open-source** release for React + Vite + Tailwind vibe coders.

## Required for v1.0

| Area | Requirement |
| ---- | ------------- |
| Onboarding | `nuvio init` + click-to-tag — no manual id required for basic edit flow |
| Overlay | Stable Simple Mode; preview/apply/undo reliable |
| Patch engine | literal + cn/clsx + conditional cn; fail-closed elsewhere |
| Libraries | shadcn + TailAdmin + DaisyUI minimum paths |
| Tooling | `doctor`, `scan`, `stats` |
| Docs | User guide, limitations, library guides, telemetry |
| Examples | `examples/` directory (see below) |
| Tests | Monorepo gate green; dogfood acceptance |
| Telemetry | Overlay funnel + CLI trend; privacy audit |

## Example projects

```text
examples/
  vite-basic/          # create-vite react-ts + init + click-to-tag demo
  shadcn-dashboard/    # shadcn card/button/table edit walkthrough
  tailadmin-demo/      # slim TailAdmin slice or pointer to dogfood app
```

# Telemetry Strategy

## Current measurement model (v0.5.5)

**Do not use one combined funnel** — CLI and overlay have different anonymous IDs.

### Primary funnel (product health)

```text
overlay_connected → first_selection → preview_changes → apply_to_code
```

**Main activation KPI:** `apply_to_code`

### CLI trend (distribution)

Track weekly **unique users** on:

```text
nuvio_cli_invoked
```

Optional secondary funnel:

```text
nuvio_cli_invoked → nuvio_init_started → nuvio_init_completed
```

### npm downloads

Downloads measure package resolution, **not** CLI runs or product usage. Use `nuvio_cli_invoked` for real CLI activity.

## Future telemetry ideas (post-v1.0)

| Idea | Value |
| ---- | ----- |
| CLI ↔ overlay identity linking | True end-to-end funnel |
| `tag_element_*` events (v0.6) | Measure click-to-tag adoption |
| `doctor_run` / `scan_run` (v0.9) | Setup friction signals |
| `classNameMode` distribution | Prioritize Tailwind engine work |
| Session duration to first `apply_to_code` | Time-to-value |

## Privacy guarantees

Telemetry is **anonymous**, **opt-out**, and **invisible** during normal use.

### Collected (examples)

- Event names
- nuvio version
- OS, arch, Node version (CLI)
- Coarse enums: `command`, `error_code`, `reason`, `result_tier`
- Aggregate counts (stats/doctor)

### Never collected

- Source code
- File contents
- File paths
- Project names
- Package names
- `data-nuvio-id` values
- className strings
- Element text
- Emails
- Usernames
- Personal data

### Disable

```bash
NUVIO_TELEMETRY=0
```

```js
localStorage.setItem("nuvio.telemetry", "0")
```

Full spec: [PostHog_telemetry.md](PostHog_telemetry.md)

# Architecture

## High-level data flow

```text
Browser (overlay)
  ↕ WebSocket
Vite dev server (@nuvio/vite-plugin)
  ↕ read/write
Source files (.tsx)
  ↕ AST
@nuvio/ast-engine
```

CLI (`@nuvio/cli`) wires the plugin + overlay at init; does not participate in edit loop at runtime.

## CLI architecture

| Module | Role |
| ------ | ---- |
| `cli.ts` | Arg parse, routing, telemetry lifecycle |
| `init.ts` | Detect project, patch vite/app/main, install packages |
| `detect-project.ts` | Preflight: React, Vite, Tailwind |
| `patch-*.ts` | Codemods for vite config, app shell, styles |
| `telemetry.ts` | PostHog node client, anonymous id |
| `doctor/scan/stats` (v0.9) | Project introspection |

## Overlay architecture

| Module | Role |
| ------ | ---- |
| `NuvioDevShell.tsx` | WebSocket client, selection, patch ack |
| `PropertyPanelShell.tsx` | Simple Mode task UI |
| Task routers | Card, Table, Button, Form, Nav, Chart, Section |
| `telemetry.ts` | PostHog JS, session guards |
| Selection guides | Onboarding, duplicate id warnings |

Dev-only gate: `import.meta.env.DEV`.

## Vite plugin architecture

| Module | Role |
| ------ | ---- |
| `index.ts` | Plugin hook registration |
| `source-index.ts` | Scan `data-nuvio-id`, duplicates, metadata |
| `nuvio-dev-session.ts` | WebSocket server, RPC handlers |
| RPC | preview, apply, undo, (future) tagElement |

## AST patch engine architecture

| Module | Role |
| ------ | ---- |
| `apply-patch.ts` | Parse → locate host by id → apply ops → generate |
| `tailwind-whitelist.ts` | Allowed utility tokens |
| Op kinds | `setText`, `mergeTailwindClassName`, `removeTailwindClassName`, layout ops |
| `ClassNameMode` | literal, cn-basic, (future) cn-conditional, cva |

Fail-closed: unsupported patterns throw readable errors, never silent writes.

## Telemetry architecture

| Surface | Client | ID storage |
| ------- | ------ | ---------- |
| CLI | posthog-node | `~/.nuvio/telemetry.json` |
| Overlay | posthog-js | PostHog localStorage persistence |

Flush: CLI shutdown + SIGINT; overlay fire-and-forget with posthog-js queue.

## Future: click-to-tag architecture (v0.6)

```text
Overlay click (no id)
  → tagElement RPC
  → ast-engine insertDataNuvioId
  → write file
  → reindex
  → overlay refresh selection
```

## Future: component-library architecture (v0.8)

```text
source-index + import scan
  → library registry match
  → host metadata (component kind, className mode)
  → overlay task router picks UI module
```

# Roadmap Prioritization

Ordered by **impact on vibe-coder time-to-first-apply**:

| Order | Milestone | Why first |
| ----- | --------- | --------- |
| 1 | **Click-to-tag (v0.6)** | Biggest onboarding blocker; removes manual ids |
| 2 | **Tailwind support (v0.7)** | Real projects use `cn()` / conditionals; without this, apply fails on AI-generated code |
| 3 | **Component libraries (v0.8)** | shadcn/TailAdmin are what users actually run; generalizes dogfood |
| 4 | **Doctor/scan/stats (v0.9)** | Supportability; lowers churn when setup is wrong |
| 5 | **Stabilization (v1.0)** | Examples, docs, tests, npm 1.0.0 — ship confidence |

**Dependency chain:** v0.8 benefits from v0.6 (tagging) + v0.7 (className modes). v0.9 can parallelize late v0.7. v1.0 is a gate, not a feature dump.

# Known Non-Goals

nuvio v1.0 will **not** target:

| Non-goal | Reason |
| -------- | ------ |
| Vue, Angular, Svelte | Focus: React + Vite + Tailwind |
| Production editing | Dev-only by design; no prod bundle |
| Hosted cloud editing | Local-first; no account/server requirement |
| Visual page builders | Not Webflow; edits real source |
| Figma replacement | Not a design tool |
| Full IDE / replace Cursor | Complement AI coding, not replace |
| npm package patching | Never patch `node_modules` |
| Backend / API editing | UI layer only |

**Stay focused:** React + Vite + Tailwind, localhost, source-backed patches.
